# Due 下一阶段产品升级与院校考情本地监控 Spec

## 背景

`Due` 当前已完成原始 MVP 主线：极简考试倒计时、复习天数、Android 桌面 Widget，以及对应的本地测试、APK 构建与启动稳定性修复。本轮不再继续补做 MVP 收尾，而是在既有实现稳定的前提下，规划一次产品定位升级。

新的产品定位从“极简考试倒计时 + 复习天数 + Android Widget”升级为“面向考试备考的倒计时 + 信息监控 + 关键节点提醒工具”。本轮新增功能主题为“院校考情本地监控”。

本 spec 以真实代码状态、`issues/due-mvp-progress-alignment.csv` 与交接摘要为基线。如果 `docs/PROGRESS.md` 与代码现状不一致，以代码和已完成 CSV 为准，不重复规划已经完成的 MVP 主线能力。

## 真实基线

以下能力已完成，不应重新作为下一阶段 MVP 待办：

- 倒计时、复习开始日期、Widget 配置三类现有数据模型、仓储、Provider 与页面主链路已可用。
- `flutter test` 已通过，当前记录为 16/16。
- `flutter build apk` 已通过，APK 已可安装到 BlueStacks 并启动。
- Android Widget 最小链路已完成并有验证记录。
- WorkManager 启动崩溃热修复已落地。

其中必须长期保护的技术边界包括：

- Android Widget 现有展示与点击回到应用链路。
- Hive 现有 `countdowns`、`review_start`、`widget_config` 三个 Box 及清空数据逻辑。
- 现有 `go_router` 路由：`/`、`/add`、`/edit/:id`、`/review-start`、`/widget-preview`、`/settings`。
- `AndroidManifest.xml` 中对 `androidx.work.WorkManagerInitializer` 的移除；后续不得恢复旧的自动初始化路径。

## 目标

在不引入服务器、VPS、无头浏览器或实时抓取承诺的前提下，为 `Due` 规划并拆解“院校考情本地监控”功能，形成一份可直接交给 Codex 执行的 spec 与 CSV。最终功能分两阶段推进：

- 阶段 A：提供“手动可用”的本地监控闭环。
- 阶段 B：在阶段 A 数据模型稳定后，再增加 Android 端低频后台检查与本地通知。

## 非目标

本轮明确不做以下内容：

- 不重做已完成的 MVP 主线功能，也不重新规划 Task 11-18 的旧收尾事项。
- 不引入服务端存储、VPS、中转 API、账号体系或云同步。
- 不做无头浏览器、登录态采集、JS 重度动态站点适配或“实时监控”承诺。
- 不扩展到 iOS Widget、跨平台后台调度或复杂消息中心。
- 不在本轮规划中修改业务代码；本轮只产出规划文档和执行 CSV。

## 用户问题定义

面向考试备考场景，用户除了需要看距离目标考试还有多少天，还需要持续关注院校官网、公告页或 RSS 中与报名、考试安排、复试、调剂、录取等相关的信息更新。当前用户往往手动反复刷新多个页面，容易漏看关键通知。

因此 `Due` 下一阶段需要提供一个“本地、低承诺、可解释”的信息监控能力：

- 用户自己维护监控源。
- App 负责按规则抓取静态网页或 RSS。
- App 根据关键词识别命中内容并保存记录。
- 抓取失败时不伪装成功，而是明确提示用户手动打开官网继续查看。

## 范围与约束

### 产品范围

阶段 A 仅覆盖以下能力：

- 用户手动添加监控对象，字段至少包括院校名称、目标 URL、关键词。
- 支持手动触发检查。
- 优先支持 RSS；对普通网页仅支持静态 HTML 文本抓取与关键词匹配。
- 保存命中记录，至少包含标题、链接、摘要、命中关键词、发现时间、来源监控对象。
- 对重复命中做去重，避免同一条记录重复堆积。
- 抓取失败时允许用户直接打开来源页面。

阶段 B 在阶段 A 基础上扩展：

- 使用 WorkManager 做 6-12 小时低频后台检查。
- 发现新命中时触发本地通知。
- 避免对同一命中重复提醒。
- 保持应用启动稳定，不破坏既有 WorkManager 热修复。

### 技术约束

- 继续沿用现有 `models/`、`repositories/`、`providers/`、`services/`、`pages/`、`router/` 分层。
- 仅在现有 Hive 体系上新增 Box，不替换现有本地存储方案。
- 所有新增路由都必须在不破坏现有路由入口和返回链路的前提下接入。
- 若引入网络依赖，应优先选择 Flutter 端易维护、可测试、无需原生桥接的最小集合。
- WorkManager 重新接入时必须采用显式初始化/配置策略，不得恢复 `WorkManagerInitializer` 自动启动路径。

### 降级说明

该功能是“本地辅助监控”，不是“官方实时推送”或“全站抓取器”。以下情况属于预期降级，而不是 bug：

- 站点内容主要依赖前端 JS 渲染，导致静态抓取无法拿到正文。
- 页面返回频率限制、反爬拦截或临时不可达。
- 页面结构变化导致摘要提取退化，但链接仍可打开。
- 某些源只能建议用户改用 RSS 或手动打开官网。

## 建议的数据与结构扩展

### 新增数据模型

建议至少新增以下模型：

- `MonitorSource`
  - `id`
  - `schoolName`
  - `sourceName` 或可选备注
  - `url`
  - `sourceType`，至少区分 `rss` 与 `webPage`
  - `keywords`
  - `isEnabled`
  - `lastCheckedAt`
  - `lastStatus`
  - `createdAt`
  - `updatedAt`

- `MonitorHit`
  - `id`
  - `sourceId`
  - `title`
  - `link`
  - `summary`
  - `matchedKeywords`
  - `publishedAt` 或 `discoveredAt`
  - `contentFingerprint`
  - `createdAt`

- `MonitorCheckResult` 或等价的运行态结构
  - `sourceId`
  - `status`
  - `checkedAt`
  - `newHitCount`
  - `errorMessage`

### 新增持久化点

建议新增 Hive Box，但要显式评估和更新 `clearAll()` 逻辑，避免只清空旧数据或误伤现有数据。

候选 Box：

- `monitor_sources`
- `monitor_hits`
- 如有必要再加 `monitor_runtime` 或把运行态留在内存状态中

## 页面与入口建议

为了减少对现有首页信息密度的破坏，建议采用“从首页进入新的监控列表页”的方式，而不是把监控结果直接塞进当前首页主列表。

建议的最小页面集合：

- `MonitorListPage`
  - 显示监控对象列表
  - 支持新增、编辑、手动检查、查看最近状态
  - 提供进入命中记录页入口

- `MonitorEditPage`
  - 新增/编辑监控对象
  - 校验 URL、关键词、来源类型

- `MonitorHitsPage`
  - 查看命中记录
  - 支持按监控对象筛选
  - 支持打开原始链接

入口建议：

- 首页增加一个明确入口卡片或按钮，进入监控列表页。
- 设置页可保留辅助入口，但不应作为唯一入口。

## 任务拆分

### Phase 0：依赖与边界校准

1. 明确抓取与解析依赖的最小集合，优先支持 RSS 和静态 HTML。
2. 明确新增数据模型、Box 名称、去重键规则、`clearAll()` 更新策略。
3. 明确新路由与页面入口方案，避免挤压现有首页主信息。

### Phase A：院校考情本地监控 MVP

4. 新增监控源与命中记录的数据模型、仓储与 Provider。
5. 实现手动检查服务：抓取、解析、关键词匹配、命中去重、结果落库。
6. 实现监控列表页、添加编辑页、命中记录页。
7. 接入首页/设置页入口与失败兜底路径。
8. 为阶段 A 的核心逻辑补测试并跑通 `flutter test`。

### Phase B：低频后台检查与本地通知

9. 在不破坏热修复的前提下，显式接入 WorkManager 初始化与调度桥接。
10. 实现后台检查与命中去重提醒策略。
11. 接入 Android 本地通知、权限与点击行为。
12. 完成 `flutter build apk`、人工后台验证与回归检查。

## 原子化执行建议

后续 CSV 拆分时应遵守以下规则：

- 先完成阶段 A 的数据层和服务层，再进入页面层。
- 把“抓取解析”和“去重落库”拆成独立 issue，便于单独验证。
- 把“页面入口接入”与“监控业务页面”拆开，降低一次改动的回归面。
- 阶段 B 单独成组，不应与阶段 A 混做。
- 每条 issue 都要写清对 Android Widget、Hive 旧 Box、既有路由、WorkManager 热修复的回归要求。

## 验收标准

### Spec 层验收

- spec 明确标注旧 MVP 已完成，不重复规划已收口事项。
- spec 明确写出两阶段范围、非目标、技术边界和降级说明。
- spec 明确列出必须保护的四类非回归边界：Widget、Hive 现有数据、既有路由、WorkManager 热修复。

### 阶段 A 目标验收

- 用户可新增、编辑、禁用监控对象。
- 用户可手动触发检查，且结果明确区分“有新命中”“无新命中”“抓取失败”。
- 命中记录可持久化、可去重、可查看、可打开原始链接。
- RSS 与静态网页至少各有一条可验证路径或测试替身。
- `flutter test` 通过，且覆盖阶段 A 的关键逻辑与至少一条页面主链路。

### 阶段 B 目标验收

- WorkManager 后台任务可按 6-12 小时低频执行。
- 新命中只提醒一次或按去重策略提醒，不产生明显重复通知。
- Android 应用冷启动、热启动和加装 Widget 均不因 WorkManager 接入再次崩溃。
- `flutter build apk` 通过，并有人工后台验证记录。

## 风险

- 当前项目尚未引入网络抓取依赖，依赖选择不当会放大维护成本或原生兼容问题。
- 不做无头浏览器意味着动态站点覆盖率有限，必须在产品文案和交互中提前说明能力边界。
- 页面解析策略若过于激进，容易因为网页结构变化频繁失效；若过于保守，又可能抓不到有效摘要。
- 新增 Hive Box 后，`clearAll()`、迁移和旧数据兼容处理若设计不清，可能影响现有倒计时与 Widget 配置。
- WorkManager 第二阶段若重新采用自动初始化，可能直接回归到已修复的启动崩溃问题。

## 回滚方案

- 规划层回滚：直接回退本 spec 与对应 CSV，不影响现有业务代码。
- 执行层回滚：要求 Codex 按 atomic issue 独立提交与验证，单条 issue 失败时只回滚该 issue 触及文件。
- 阶段 B 回滚：若后台调度或通知出现稳定性问题，应保留阶段 A 手动检查闭环，整体禁用自动检查与通知入口，不影响监控数据本身。
- 数据层回滚：新增 Box 如需停用，应提供明确的停用策略，避免误删已有 `countdowns`、`review_start`、`widget_config` 数据。
